# OA Mobile Lab — desktop and Android validation

*Reference sweep: 2026-07-18 · signed release APK · all five canonical Byte routes passed.*

OA Mobile Lab exercises the same C++ operation library, autograd engine, AdamW
optimizer, `.oam` checkpoint format, and SPIR-V training kernels used by OA on desktop.
This is a controlled physical-device acceptance record, not a claim that a small phone
matches a laptop GPU or that every Android driver is supported.

## Controlled workload

Both platforms use the canonical `OaNlpSuite` contract:

- 300 optimizer updates, batch 64, sequence length 16;
- 1,024 predicted positions per update;
- the same 576-byte teaching corpus and initialization seed;
- width 32, recurrent/FFN width 64, and a 256-byte vocabulary;
- prompt `to be`, followed by 80 autoregressively generated bytes;
- FP32 forward, cross entropy, backward, AdamW, evaluation, generation, `.oam`
  save/load, and exact generation parity after fresh-model reload.

The gate rejects a route unless it completes every stage, reaches its learning threshold,
generates a valid corpus continuation, covers at least 90% of corpus eight-grams, and
reproduces the generated bytes after checkpoint reload. A low teacher-forced loss alone
is not a pass.

## Results

Desktop values are the three-process means published in
[the complete NLP benchmark](OaNlpSuite.md). Mobile values are one uninterrupted clean
process per architecture from the signed release package. They are functional references,
not a statistically averaged mobile silicon benchmark.

| Architecture | Params | Desktop wall ms/step | Mobile wall ms/step | Desktop GPU ms/step | Mobile GPU ms/step | Mobile byte/s |
|---|---:|---:|---:|---:|---:|---:|
| RNN | 31,104 | 5.00 | 154.06 | 4.482 | 147.52 | 6,646.87 |
| GRU | 43,648 | 11.17 | 250.15 | 10.498 | 237.54 | 4,093.46 |
| Transformer | 25,760 | 7.99 | 263.11 | 7.033 | 254.08 | 3,891.89 |
| Sparse MoE Transformer | 28,068 | 8.62 | 221.31 | 7.514 | 211.35 | 4,626.92 |
| Mamba-3 | 25,800 | 33.34 | 1,103.00 | 32.319 | 1,081.47 | 928.38 |

| Architecture | Final CE | Accuracy | Checkpoint round trip | Generation quality |
|---|---:|---:|---|---|
| RNN | 0.1877 | 92.3% | Pass | Pass, 100% corpus eight-grams |
| GRU | 0.1758 | 92.3% | Pass | Pass, 100% corpus eight-grams |
| Transformer | 0.1943 | 92.5% | Pass | Pass, 100% corpus eight-grams |
| Sparse MoE Transformer | 0.1907 | 93.2% | Pass | Pass, 100% corpus eight-grams |
| Mamba-3 | 0.2004 | 92.7% | Pass | Pass, 100% corpus eight-grams |

At this deliberately tiny shape, sparse MoE took **0.84x** the dense Transformer's
mobile wall time and **0.83x** its GPU time in this controlled run. That is evidence that
the GPU-authored sparse path is viable on Adreno; it is not evidence of a general MoE
quality or performance advantage across model sizes.

## Cross-vendor defect found by the gate

The first signed-release sweep exposed corrupted Transformer learning on Turnip while
the same packed projection passed on desktop. The packed Q/K/V shader selected bindless
descriptors from a thread-varying projection lane without the required non-uniform
contract. The release fix uses three explicit uniform branches while retaining one
dispatch. The repeated signed-release run then passed both dense Transformer and MoE.

This is why the physical phone gate is part of the release contract: desktop success is
not treated as proof that descriptor behavior is portable.

## Reproduce

Build a locally signed package using the environment contract documented in
[the Mobile Lab README](../../Apps/Android/OaMobileLab/README.md), install it, then run:

```bash
OA_ANDROID_PACKAGE=com.oa.mobilelab \
OA_NLP_TIMEOUT_SECONDS=1800 \
  Apps/Android/OaMobileLab/tools/run-nlp-suite.sh
```

The runner discovers whether the package is debuggable. Debug builds expose the native
report through the app-private filesystem; release builds emit the same bounded report
through the `OaMobileReport` log tag. Both paths validate steps, cancellation state,
checkpoint parity, generation quality, prompt, and generated output.

## Scope

- This record covers one Adreno 610 phone and one app-local Turnip version.
- Android recurrent and Mamba-3 backward paths use bounded mobile kernels where the
  larger desktop fused scans exceed the target compiler's practical limits.
- Mobile results can vary with temperature, power policy, and foreground scheduling.
- The teaching corpus validates execution and learning behavior; it does not represent
  production language-model quality.
